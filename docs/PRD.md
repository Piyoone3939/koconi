# PRD: Koconi — 旅行記録・写真管理・マップアプリ

## 概要
Koconiは、ユーザーが行った場所と写真を直感的に記録・視覚化できるモバイルアプリです。写真撮影モードでは、撮影した写真からAIでランドマークを検出し、それを地図上に”建てる”ことで、タップ時にその写真・コメント・位置情報が表示されます。共有は全世界公開ではなく、招待ベースのグループまたは個人間で限定されます。

### 目的
- 写真から自動でランドマークを抽出して地図上に配置し、視覚的な思い出のナビゲーションを提供する。
- プライバシー重視の共有モデル（招待制）を実装する。

### 成功基準（KPI）
- 月間アクティブユーザー（MAU）: 10,000（ローンチ12ヶ月以内の目標）。
- 写真からの自動ランドマーク抽出の正答率（ユーザーが"正しい"とマークした割合）: 85%以上。
- 共有招待によるグループ作成数: 2,000（ローンチ6ヶ月）。
- ユーザーあたり平均保存写真数: 50枚（ローンチ6ヶ月）。

## ユーザーとペルソナ
- ペルソナ A: 旅行好きな個人（10–50代）。旅行ルートを振り返り、写真と場所を関連付けて保存したい。
- ペルソナ B: 家族や友人グループ。旅行写真を限られたメンバー間で共有・記録したい。
- ペルソナ C: フォトグラファー／趣味で写真を撮る人。撮影位置とランドマーク情報を詳しく残したい。

## 対象プラットフォーム
- iOS（iPhone）
- Android
- 将来的なWebビューワ（読み取り専用、共有閲覧用）

## コア機能

1. 写真撮影とランドマーク抽出
   - アプリ内カメラモードで写真撮影（位置メタデータを自動付与）。
   - 写真アップロード時にAIモデル（クラウドまたはオンデバイス）でランドマーク検出。
   - 検出されたランドマーク候補(ローポリ)を地図上にピン/オブジェクトとして配置。
   - ユーザーがランドマーク名を編集・承認できるワークフロー。
   - ランドマークは写真とリンクされ、タップで写真・コメント・撮影日時を表示。

2. アルバムページ
   - 旅行単位・日付・場所・タグでフィルタリングできる写真アルバム。
   - サムネイル表示・タップで拡大表示・コメント追加。
   - バッチダウンロード（ユーザーの端末またはクラウド）や共有用リンク生成（招待制での閲覧に限定）。

3. 共有とプライバシー
   - 招待ベースのグループ（グループ所有者・管理者・メンバー）をサポート。
   - グループ内での閲覧・コメント・写真追加権限のロール管理。
   - 共有リンクは必須でトークン・有効期限・アクセス権を持つ。

4. 検索・フィルタ
   - 日付・場所名（AI抽出名を含む）・タグ・グループで検索。
   - 地図上のズームレベルに基づくランドマーククラスター表示。

5. 設定と通知
   - データ同期のオン/オフ（モバイルデータでの同期制御）。
   - 共有された新規写真・コメントのプッシュ通知（グループ単位で設定可能）。

6. 360°シーンビューア（Premium機能）
   - 3Dピン配置時に、周囲4方向（前後左右）をガイド付きで撮影し「シーン」として保存する。
   - 地図上のシーンありピンをタップすると、撮影当時の360°環境をスマホを動かして閲覧できる。
   - ジャイロセンサー対応により、端末を向けた方向の映像が表示される（その場にいるような没入感）。
   - シーンの記録はPremiumプラン加入者のみ可能。閲覧は無料ユーザーにも開放し、拡散を促す。
   - 技術方式: 複数枚方式（4方向 = N/E/S/W）。Expo Managed Workflow 上で WebView + Three.js により実装。

## 非機能要件
- スケーラビリティ: ユーザーデータは最初は限定的（数万ユーザー）を想定するが、将来的に100万ユーザー規模へ水平スケールできる構成。
- 可用性: SLA 99.9%（主要API）。
- セキュリティ: データは転送中はTLS、保存時は暗号化（サーバー側のストレージ暗号化）。
- レイテンシ: AIランドマーク抽出の平均処理時間は1枚あたり500ms〜3s（クラウド）を目標。

## UXフロー（主要シナリオ）

### 新規ユーザーのオンボーディング
1. アプリ起動 → アカウント作成（メール/パスワード、またはOAuth: Google/Apple）。
2. チュートリアル: 地図の見方、写真撮影モード、共有グループの作成方法。

### 旅行記録の作成
1. アプリ内カメラモードで写真撮影（位置メタデータを自動付与）。
  - 写真アップロード時にAIが撮影画像を解析し、内部に予め組み込まれたローポリ（低ポリゴン）3Dランドマーク資産群から最適候補を推定する。
  - 検出は「候補がどの内蔵3Dモデルに最も近いか」を返す形式となり、AIは候補モデルのID、信頼度（match_score）、推奨スケール・回転などの配置ヒントを返す。
  - ユーザーには推奨候補の3Dサムネイル（回転アニメーション可）をカルーセルで提示し、ユーザーが最も適切なローポリモデルを選択して地図上に配置するワークフロー。
  - 選択後は地図上に3Dモデル（glTF/GLB等）として表示。タップでそのランドマークに紐づく写真・コメント・撮影日時・位置精度・モデル情報（モデルID, バージョン）を表示。
  - ユーザーはモデルの微調整（位置・回転・スケール）、名称編集、または「写真に適したモデルが見つからない」場合は手動で非該当／新規タグ付けできる。
2. 写真・コメント・日時・撮影位置のズーム表示。
3. 編集（タイトル変更、写真追加/削除）、共有、削除オプション。
## AIランドマーク抽出要件（ローポリ3Dマッチング中心）
- 入力: JPEG/HEIC/PNG等の写真、付随する位置情報（もしあれば）。
- 出力: 内蔵3D資産IDの候補リスト（asset_id, match_score, suggested_scale, suggested_rotation, confidence）および（任意）補助のテキスト候補（name, category, wiki_url）。
- モード: 主にオンデバイスでのマッチング優先（アプリにローポリ資産をバンドルまたは初回ダウンロード）。ネットワークがある場合はクラウドで高精度マッチングを行い、内部資産IDへの候補マッピングを返す。オフライン時はオンデバイスの軽量マッチャで候補を生成。
- 学習データ: 内蔵資産それぞれに複数の視点画像を用意し、モデルは画像特徴量と資産IDの紐付けを学習する。公開ランドマークデータやユーザーのオプトインデータを利用してモデルを強化。
- インタラクション: ユーザー承認（モデル選択）と編集のフィードバックはマッチング精度改善に使用（オプトイン）。ユーザーが承認したペアは学習データとして匿名化して収集可能。
- プライバシー: ユーザーが写真を外部（クラウド）に送るか選択できる。オンデバイス優先の設定を推奨（デフォルトはオンデバイスマッチング、ユーザーがクラウド送信を許可した場合のみアップロード）。

## AIランドマーク抽出要件
- 入力: JPEG/HEIC/PNG等の写真、付随する位置情報（もしあれば）。
- 出力: ランドマーク候補のリスト（name, confidence, bounding_box?, category, wiki_url?）
- モード: クラウド（高精度・高速）とオンデバイス（オフライン時の簡易候補）をサポート。
- 学習データ: 旅行写真・ランドマークデータセット（公開データ + ユーザー提供オプトインデータ）。
- インタラクション: ユーザーによる候補承認/修正のフィードバックを集め、モデルの継続学習に利用（オプトイン）。
- プライバシー: ユーザーがAIに写真を送るかどうか選択できる設定。

## 認証・権限モデル
- OAuth2 / JWT ベースの認証。
- リソースレベルのACL: trips/photos/landmarksはuserまたはgroup所有、グループ内ロールによるアクセス制御。
- 招待はトークン付きURLまたはメール招待。

## ストレージ設計
- メタデータ: RDBMS（Postgres）
- 写真アセット: オブジェクトストレージ（S3互換）
- キャッシュ: CDN（写真）、Redis（セッション、クエリキャッシュ）

## インフラ（高水準）
- API: コンテナ化（Docker）→ Kubernetes
- DB: マスター/リードレプリカ（Postgres/PostGIS）
- オブジェクトストレージ: S3（AWS/GCP/MinIO）
- AI: 推論サーバー（GPUインスタンス, Dockerized）, キュー（RabbitMQ / SQS）で非同期処理
- CI/CD: GitHub Actions → Canary / Blue-Green deploy

## 監視・ログ

## ロードマップ（フェーズ別）
- M0（準備）: 仕様固め, MVP設計, データモデル, AI選定
- M1（MVP）: 写真撮影・ランドマーク抽出、アルバム、招待ベース共有、AIサーバーのPoC
- M2: オンデバイスAI追加、フィードバックループによるAI改善
